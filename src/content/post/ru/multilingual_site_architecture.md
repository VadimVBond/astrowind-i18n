# 1. Контекст

![Статический сайт](https://images.unsplash.com/photo-1554130916-c2aebf8c8770)

Цель — создание портфолио-проекта и инженерного блога без использования CMS.

Требования:

- Полная статическая генерация
- Поддержка нескольких языков
- Контент на Markdown
- Контроль над структурой URL
- Минимизация runtime-логики
- Предсказуемая SEO-архитектура

Стек:

- MkDocs для документационных разделов
- Astro для портфолио и блоговой части
- Tailwind / Bootstrap в зависимости от задачи

Проект проектировался как расширяемый и CDN-совместимый.

---

# 2. Проблема

![CMS vs Static](https://images.unsplash.com/photo-1559526324-593bc073d938)

Большинство мультиязычных сайтов строятся на CMS:

- WordPress + плагины
- Headless CMS
- Strapi / Contentful

Проблема такого подхода:

- Избыточность для портфолио
- Усложнение инфраструктуры
- API-зависимость
- Повышенная точка отказа
- Runtime-зависимость

Задача: реализовать мультиязычность без базы данных и без административной панели.

---

# 3. Ограничения

![Ограничения](https://images.unsplash.com/photo-1581091870629-33b3c9f2c55e)

- Полная статическая генерация (SSG)
- Отсутствие сервера приложений
- SEO-совместимость
- Возможность хостинга на CDN
- Контроль canonical и hreflang
- Минимизация дублирования структуры

# 4. Рассмотренные варианты

## Вариант A — Папочная сегрегация

```
/en/
/ru/
/uk/
```

Плюси:
- Простая логика
- Явное разделение

Минусы:
- Дублирование структуры
- Повышенный риск рассинхронизации
- Рост поддерживающей сложности

Отказ: высокая стоимость сопровождения при масштабировании.

## Вариант B — Headless CMS

Плюси:
- Управление через UI
- Централизация данных

Минусы:
- Усложнение архитектуры
- API-зависимость
- Потеря статической простоты
- Избыточность для портфолио

Отказ: неоправданная сложность.

## Вариант C — Content collections с lang-полем (выбранный)

```
src/content/blog/article.ru.md
src/content/blog/article.en.md
```

Плюси:
- Единый layout
- Общий slug
- Централизованный SEO-контроль
- Простое масштабирование

Минусы:
- Требуется дисциплина в naming-конвенциях
- Нет автоматической синхронизации переводов

![Контент коллекции](https://images.unsplash.com/photo-1590608897129-79cfef5b9f3f)

# 5. Выбранный подход

Ключевая идея — язык как атрибут контента, а не как структурная папка.

Каждая статья:
- имеет единый slug
- отличается только `lang`
- генерируется в пространстве `/ru/slug`

Это позволяет:
- централизовать SEO
- управлять hreflang
- не дублировать layout
- сохранить статическую простоту

![SEO и hreflang](https://images.unsplash.com/photo-1537432376769-00a4c61f2010)

# 6. Детали реализации

## Структура

```bash
src/
  content/
    blog/
      multilingual-architecture.ru.md
      multilingual-architecture.en.md
```

## Frontmatter

```yaml
lang: ru
slug: multilingual-architecture
```

## Генерация маршрутов
- Получение всех статей
- Группировка по slug
- Генерация alternate-ссылок

## SEO
- Canonical для каждой языковой версии
- Автоматическая генерация hreflang
- Исключение draft из индексации

## Навигация
- Автоматическая генерация на основе content collections
- Без ручного дублирования структуры

![Маршруты и навигация](https://images.unsplash.com/photo-1557804506-669a67965ba0)

# 7. Компромиссы (Trade-offs)

1. Нет административного интерфейса
2. Нет live-переводов
3. Нет runtime-переключения языка
4. Требуется дисциплина структуры

Архитектура упрощает инфраструктуру, но требует инженерной строгости.

# 9. Выводы и уроки

Мультиязычность не требует CMS.

Если:
- объём контента ограничен
- нет требований к runtime-редактированию
- важна предсказуемость архитектуры

Статический подход оказывается устойчивее.

Ключевой фактор — дисциплина в организации контента и slug-конвенциях.
