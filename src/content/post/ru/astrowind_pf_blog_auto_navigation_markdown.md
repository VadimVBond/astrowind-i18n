# 1. Контекст

В проектах на основе Markdown (документация, технические блоги, портфолио, внутренние базы знаний) навигация часто формируется вручную. При масштабировании структуры — добавлении языков, разделов, вложенных категорий — поддержка меню становится отдельной задачей.

При наличии нескольких языков, повторно используемых компонентов и нескольких типов страниц (статьи, кейсы, документация) ручная синхронизация навигации перестаёт быть устойчивым решением.

Цель — сформировать архитектуру, при которой структура проекта становится источником правды, а навигация и вспомогательные данные генерируются автоматически.

---

# 2. Проблема

Основные симптомы:

- Дублирование структуры в конфигурациях.
- Разхождение между файловой системой и меню.
- Ручная синхронизация при добавлении языков.
- Рост числа ошибок при рефакторинге.
- Потеря консистентности URL.

В многоязычных проектах проблема усиливается: одна и та же иерархия должна поддерживаться в нескольких директориях.

---

# 3. Ограничения

- Markdown остается основным источником контента.
- Не допускается ручная правка навигации в нескольких местах.
- Структура должна быть масштабируемой.
- Поддержка SEO (читаемые URL, стабильность ссылок).
- Минимальная зависимость от внешних плагинов.
- Возможность интеграции AI в процесс генерации и проверки.

---

# 4. Рассмотренные варианты

## Вариант 1: Полностью ручная конфигурация

Навигация описывается в YAML/JSON конфигурации.

Плюсы:
- Полный контроль порядка.

Минусы:
- Дублирование.
- Высокая вероятность рассинхронизации.
- Низкая масштабируемость.

Решение отклонено из‑за высокой операционной нагрузки.

---

## Вариант 2: Автоматическая генерация на основе файловой структуры

Меню строится путём сканирования директорий.

Плюсы:
- Нет дублирования.
- Файловая система — источник правды.

Минусы:
- Потеря контроля над порядком.
- Ограниченная гибкость.

Решение принято частично.

---

## Вариант 3: Метаданные в Frontmatter + генератор навигации

Каждый Markdown‑файл содержит структурные параметры:

- order
- section
- lang
- slug

Генератор формирует дерево на основе метаданных.

Плюсы:
- Контроль порядка.
- Поддержка мультиязычности.
- Расширяемость.

Минусы:
- Требуется дополнительный слой обработки.

Выбран как основной подход.

---

# 5. Выбранный подход

Архитектура состоит из трёх уровней:

1. Файловая структура — базовая иерархия.
2. Frontmatter — декларативные метаданные.
3. Генератор навигации — скрипт, формирующий структуру.

Файловая система определяет разделы, а метаданные — порядок и логическую группировку.

Структура проекта:

```
content/
  en/
    blog/
    docs/
  ru/
    blog/
    docs/
scripts/
  generate-navigation.ts
```

Генератор:
- Сканирует директории.
- Читает frontmatter.
- Строит дерево.
- Экспортирует JSON.

---

# 6. Детали реализации

## 6.1. Формат Frontmatter

```
---
title: "Статья"
order: 2
section: "architecture"
lang: "ru"
slug: "auto-navigation"
---
```

## 6.2. Алгоритм генерации

1. Рекурсивное сканирование каталога.
2. Группировка по языкам.
3. Группировка по section.
4. Сортировка по order.
5. Генерация структуры:

```
{
  "ru": {
    "architecture": [ ... ]
  }
}
```

## 6.3. Интеграция в сборку

- Скрипт запускается перед сборкой.
- JSON импортируется в layout.
- Меню формируется динамически.

## 6.4. AI в workflow

- Проверка консистентности метаданных.
- Поиск отсутствующих slug.
- Анализ конфликтов order.

AI используется как инструмент валидации, а не источник структуры.

---

# 7. Компромиссы (Trade-offs)

| Решение | Преимущество | Ограничение |
|----------|--------------|-------------|
| Автогенерация | Нет дублирования | Дополнительный слой обработки |
| Frontmatter | Контроль порядка | Требует дисциплины |
| JSON-экспорт | Чистая архитектура | Необходима синхронизация при сборке |

Главный компромисс — увеличение сложности инфраструктуры ради устойчивости и масштабируемости.

---

# 8. Результат

- Структура стала источником правды.
- Добавление нового языка не требует изменения конфигурации.
- Исключено ручное обновление меню.
- Снижено количество ошибок при рефакторинге.
- Повышена предсказуемость URL.

Навигация масштабируется линейно с ростом проекта.

---

# 9. Выводы и уроки

1. Файловая структура должна быть декларативной.
2. Метаданные должны контролировать логику, а не конфигурации.
3. Автоматизация оправдана при росте проекта.
4. AI полезен как инструмент проверки, но не как источник архитектуры.
5. Стойкость важнее локального удобства.

Автоматизация навигации — не оптимизация, а переход к управляемой архитектуре Markdown‑проектов.
