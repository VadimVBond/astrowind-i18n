# Когда Tailwind лучше Bootstrap (и наоборот)

## 1. Контекст

В рамках разработки проектов на Astro с мультиязычной архитектурой, переиспользуемыми компонентами и строгими требованиями к производительности возникает необходимость системного выбора CSS‑подхода.

Варианты — Tailwind (utility-first) и Bootstrap (component-based framework). Оба инструмента зрелые, но архитектурно ориентированы на разные задачи.

Выбор влияет на:

- структуру проекта
- повторное использование компонентов
- объем CSS
- скорость сборки
- масштабируемость дизайн‑системы
- долгосрочную поддерживаемость

Задача — определить, в каких условиях каждый инструмент является рациональным архитектурным решением.

---

## 2. Проблема

Типичная ошибка — выбирать инструмент по личным предпочтениям.

В реальных проектах критерии иные:

- требуется ли строгая дизайн‑система
- будет ли проект расти
- предполагается ли кастомный UI
- важна ли скорость прототипирования
- есть ли ограничения по SEO и performance
- сколько разработчиков будет работать над кодовой базой

Bootstrap и Tailwind по‑разному решают эти задачи.

---

## 3. Ограничения

В рамках Astro‑проекта учитывались следующие ограничения:

- Статическая генерация (SSG)
- Мультиязычность
- Разделение layout / components / blocks
- Возможность частичного переиспользования UI между проектами
- Минимизация CSS-бандла
- Отсутствие избыточного JavaScript

Дополнительно:

- Компоненты должны быть изолированы
- Должна поддерживаться темизация (dark/light)
- Нужна гибкость для кастомных лендингов

---

## 4. Рассмотренные варианты

### Вариант 1 — Bootstrap как основной UI-фреймворк

Плюсы:

- Быстрый старт
- Готовые компоненты
- Консистентная сетка
- Предсказуемая структура

Минусы:

- Избыточный CSS
- Ограниченная гибкость кастомизации
- Стили по умолчанию требуют переопределения
- Компонентная логика может конфликтовать с Astro-архитектурой

---

### Вариант 2 — Tailwind как utility-first система

Плюсы:

- Минимальный итоговый CSS (через purge)
- Высокая гибкость
- Хорошая интеграция с компонентной архитектурой
- Удобство темизации

Минусы:

- Повышенная плотность классов в разметке
- Требует дисциплины
- Нужна собственная дизайн-система

---

### Вариант 3 — Гибридная модель

Bootstrap для сетки и базовых компонентов.
Tailwind для кастомной стилизации.

Результат — дублирование ответственности и рост сложности.

Отказ от варианта по причине архитектурной неоднозначности.

---

## 5. Выбранный подход

Подход определялся типом проекта.

### Tailwind используется когда:

- Проект требует кастомного дизайна
- Есть собственная дизайн‑система
- Нужна высокая гибкость
- Требуется минимальный CSS-бандл
- Компоненты должны быть максимально переиспользуемыми

### Bootstrap используется когда:

- Нужен быстрый MVP
- Дизайн стандартный
- Проект небольшой
- Приоритет — скорость разработки
- Нет ресурсов на построение собственной системы

Таким образом, выбор стал контекстно‑зависимым.

---

## 6. Детали реализации

### Архитектура с Tailwind

```
src/
 ├── layouts/
 ├── components/
 ├── blocks/
 ├── styles/
 │    └── global.css
 └── config/
      └── tailwind.config.js
```

Ключевые принципы:

- Конфигурация design tokens в tailwind.config.js
- Использование @apply только в слое abstraction
- Темизация через class="dark"
- Разделение layout и UI-компонентов

Пример design token:

```js
// tailwind.config.js
export default {
  theme: {
    extend: {
      colors: {
        primary: '#1e293b'
      }
    }
  }
}
```

Результат — управляемая дизайн-система.

---

### Архитектура с Bootstrap

```
src/
 ├── layouts/
 ├── components/
 ├── assets/
 │    └── bootstrap.scss
```

Ключевые шаги:

- Использование SCSS-версии
- Отключение ненужных модулей
- Кастомизация переменных

Пример:

```scss
$primary: #1e293b;
@import "bootstrap/scss/bootstrap";
```

Однако итоговый CSS остается существенно больше.

---

## 7. Компромиссы (Trade-offs)

| Критерий | Tailwind | Bootstrap |
|----------|-----------|------------|
| Размер CSS | Минимальный | Значительный |
| Скорость старта | Средняя | Высокая |
| Кастомизация | Высокая | Ограниченная |
| Поддержка дизайн‑системы | Гибкая | Жесткая |
| Поддерживаемость | Требует дисциплины | Предсказуемая |
| SEO (косвенно через performance) | Лучше | Хуже |

Ключевой компромисс:

Tailwind требует архитектурного мышления.
Bootstrap снижает порог входа, но ограничивает масштабируемость.

---

## 8. Результат

В рамках Astro‑проектов с долгосрочной поддержкой и мультиязычной структурой предпочтение отдано Tailwind.

Причины:

- Контроль над CSS
- Возможность построения собственной дизайн‑системы
- Лучшая интеграция с компонентной архитектурой
- Минимизация технического долга

Bootstrap остается инструментом для:

- Прототипов
- Админ‑панелей
- Внутренних инструментов

---

## 9. Выводы и уроки

1. Нет универсального решения.
2. Выбор должен зависеть от масштаба и горизонта проекта.
3. Для долгосрочной архитектуры utility-first модель более устойчива.
4. Для быстрых решений компонентный фреймворк рационален.
5. Гибридный подход увеличивает сложность.

Фреймворк стилей — это не визуальный выбор. Это архитектурное решение, влияющее на поддержку, масштабируемость и структуру проекта.
