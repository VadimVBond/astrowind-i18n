# 1. Контекст

![Обложка технологического стека](../tech-stack-architecture-cover.svg)

В работе используется стек, ориентированный на разработку статических и гибридных сайтов, документации, а также backend-решений на Python. Основные направления включают Astro для frontend-оркестрации, MkDocs для документационных проектов, Python-фреймворки для API и AI-инструменты как вспомогательный слой для анализа и ускорения инженерного процесса.

Цель такого стека не в количестве технологий, а в управляемости архитектуры. Он должен обеспечивать масштабируемость решений, повторное использование компонентов между проектами и предсказуемую структуру при росте требований.

---

# 2. Проблема

Большинство проектов малого и среднего масштаба сталкиваются с повторяющимися трудностями:

- разрозненная структура кода;
- слабая стратегия повторного использования;
- отсутствие системной мультиязычности;
- смешение контента, логики и представления;
- низкая управляемость при росте проекта.

Дополнительно возникает задача рациональной интеграции AI-инструментов без потери архитектурного контроля. Нужен стек, который остаётся модульным, поддерживает проекты разной сложности и не заставляет менять платформу при каждом новом типе задачи.

---

# 3. Ограничения

При выборе технологий учитывались следующие ограничения:

1. Низкий порог инфраструктурных затрат.
2. Возможность деплоя на edge-платформах и статическом хостинге.
3. Минимальная зависимость от тяжёлых CMS.
4. Чёткое разделение frontend и backend.
5. Возможность интеграции AI без передачи чувствительных данных.
6. Контроль структуры проекта вручную, без скрытых абстракций.

Стек должен быть управляемым на уровне архитектуры, а не только на уровне отдельных фреймворков.

---

# 4. Рассмотренные варианты

## 4.1 Полноценный SPA-подход

Рассматривались Next.js и сходные решения.

Преимущества:

- высокая гибкость;
- зрелая экосистема.

Причины отказа:

- избыточность для большинства проектов;
- повышенная сложность инфраструктуры;
- риск неоптимального SEO при неправильной конфигурации.

## 4.2 CMS-подход

Рассматривались WordPress и аналогичные системы.

Преимущества:

- быстрый старт;
- наличие готовых плагинов.

Причины отказа:

- слабый контроль архитектуры;
- ограниченная масштабируемость структуры;
- повышенные риски технического долга.

## 4.3 Чистый backend плюс шаблоны

Преимущества:

- полный контроль над логикой.

Причины отказа:

- дублирование логики;
- слабая повторная используемость UI;
- сложности при масштабировании frontend-части.

---

# 5. Выбранный подход

Выбран гибридный архитектурный стек:

- Astro как основной frontend-оркестратор;
- MkDocs для документационных и структурированных проектов;
- Python на Django Ninja, FastAPI и Flask для API и серверной логики;
- AI-инструменты как вспомогательный инженерный слой.

Ключевая идея заключается в разделении ответственности.

| Уровень | Инструмент | Роль |
| --- | --- | --- |
| UI и контент | Astro | Статическая генерация, SEO, композиция страниц |
| Документация | MkDocs | Структурированный контент и навигация |
| API | Django Ninja / FastAPI | Лёгкий и типизированный backend |
| Лёгкие сервисы | Flask | Изолированные микросервисы и прототипы |
| AI | Gemini / Copilot / ChatGPT | Поддержка анализа, черновиков и проверки |

---

# 6. Детали реализации

## 6.1 Frontend на Astro

Базовая структура выглядит так:

```text
src/
  components/
  layouts/
  pages/
  libs/
  i18n/
public/
```

Принципы следующие:

- компоненты не зависят от конкретных страниц;
- мультиязычность строится через структуру папок и словари;
- переиспользуемые блоки выносятся в `libs/`.

## 6.2 Стратегия мультиязычности

Используются:

- структурное разделение языков (`/en/`, `/ru/`, `/uk/`, `/it/`);
- централизованные словари;
- явная маршрутизация;
- SEO-ориентированная структура URL.

Принцип простой: язык является частью архитектуры, а не надстройкой после сборки проекта.

## 6.3 Backend на Python

### Django Ninja / FastAPI

Используются для:

- JSON API;
- административных сервисов;
- интеграции с базами данных.

Выбор обусловлен:

- производительностью;
- типизацией;
- предсказуемой структурой проекта.

### Flask

Применяется для:

- лёгких сервисов;
- прототипирования;
- изолированных микросервисов.

## 6.4 Интеграция AI в workflow

AI рассматривается как инструмент повышения качества, а не как источник архитектурных решений.

Разделение ролей:

- генерация черновиков и рутинных заготовок;
- усиление аргументации и анализ архитектуры;
- техническая корректировка текста и формулировок.

AI не определяет структуру проекта. Он используется для ускорения и дополнительной проверки.

## 6.5 Повторное использование компонентов

Компоненты проектируются:

- независимыми;
- без привязки к конкретному проекту;
- с изолированными стилями;
- с возможностью вынесения в отдельные библиотеки.

Цель состоит в формировании собственной внутренней компонентной базы.

# 7. Компромиссы

## 7.1 Astro против SPA

Плюсы:

- высокая производительность;
- контроль над HTML;
- SEO-дружественная выдача.

Минусы:

- меньше интерактивности по умолчанию.

Решение: использовать islands architecture там, где это действительно необходимо.

## 7.2 Мультиязычность через структуру

Плюсы:

- явность;
- прозрачность для SEO.

Минусы:

- рост количества файлов;
- увеличение объёма поддержки.

## 7.3 Разделение backend и frontend

Плюсы:

- независимое развитие;
- масштабируемость.

Минусы:

- дополнительная координация API и контрактов.

## 7.4 AI-интеграция

Плюсы:

- ускорение разработки;
- улучшение формулировок и черновиков.

Минусы:

- риск поверхностных решений;
- необходимость постоянной инженерной валидации.

---

# 8. Результат

Сформирован стек, который:

- позволяет запускать проекты быстро;
- предсказуем в сопровождении;
- масштабируется от лендинга до сложной структуры;
- контролируем на уровне архитектуры;
- поддерживает повторное использование компонентов;
- совместим с современными AI-инструментами.

Проекты при этом сохраняют высокую производительность, чистую структуру, SEO-ориентированную организацию и готовность к расширению.

---

# 9. Выводы

1. Технологический стек должен подбираться не по моде, а по архитектурной совместимости.
2. Простота сопровождения важнее ширины экосистемы сама по себе.
3. Мультиязычность, SEO и повторное использование нужно учитывать на уровне базовой структуры.
4. AI усиливает процесс только тогда, когда уже существует дисциплина в архитектуре.
5. Хороший стек уменьшает стоимость изменений и помогает проекту расти без хаотичного усложнения.

Именно поэтому сильный стек определяется не количеством инструментов, а согласованностью их ролей внутри системы.
