# 1. Контекст

![Обложка архитектуры контента](../content-scaling-architecture-cover.svg)

Проект представляет собой мультиязычную платформу на Astro с техническим блогом, документацией и портфолио. Контент генерируется статически, поэтому структура данных влияет не только на удобство редактирования, но и на SEO, маршрутизацию, предсказуемость URL и стоимость поддержки.

Требования к модели были следующими:

- изоляция локалей без хаотичного дублирования;
- единая логика маршрутов;
- предсказуемая SEO-структура;
- возможность добавления новых языков без рефакторинга ядра;
- простая поддержка для команды без избыточных абстракций.

На раннем этапе статьи добавлялись линейно, а локализация решалась фактическим копированием файлов. Пока материалов было мало, модель казалась удобной. С ростом количества статей и языков стали видны системные ограничения.

---

# 2. Проблема

![Рост количества файлов и рассинхронизация](../content-scaling-architecture-problem.svg)

Основные сложности проявились быстро:

1. Каждый новый язык увеличивал число файлов пропорционально.
2. Навигация и списки статей могли расходиться между локалями.
3. Frontmatter дублировался и начинал расходиться.
4. SEO-поля приходилось контролировать вручную.
5. Структура проекта становилась менее предсказуемой.

Модель "одна статья в каждой локали как отдельная сущность" масштабировалась плохо, потому что усложняла не контент сам по себе, а сопровождение структуры.

---

# 3. Ограничения

Решение нужно было принимать с учётом реальных технических ограничений:

1. Только статическая генерация, без серверной логики.
2. Сохранение понятных URL для каждой локали.
3. Минимизация ручной синхронизации.
4. Возможность добавлять новые языки без переписывания маршрутов.
5. Поддерживаемость на уровне обычного инженерного процесса, без сложной CMS-модели.

Иными словами, требовалась архитектура, которая остаётся простой в эксплуатации, но не разваливается при росте числа материалов.

---

# 4. Рассмотренные варианты

## Вариант 1: Полное дублирование структуры по языкам

```text
content/
  ru/
  en/
  uk/
  it/
```

Плюсы:

- простая начальная реализация;
- визуально понятная изоляция языков.

Минусы:

- линейный рост файлов;
- высокий риск расхождения версий;
- повторение метаданных;
- ручная синхронизация структуры.

Подход оказался слишком дорогим в сопровождении.

## Вариант 2: Один источник и переводы через словари

Контент хранится в одном файле, а текстовые фрагменты переводятся через i18n-словарь.

Минусы:

- плохо подходит для больших статей;
- ограничивает редакторскую независимость локалей;
- усложняет SEO-адаптацию длинного контента;
- делает Markdown-контент неудобным для авторов.

Для полноценных инженерных статей такой вариант оказался архитектурно слабым.

## Вариант 3: Контентные коллекции Astro с разделением по языкам

Использование content collections с типизированным frontmatter и единым соглашением по именованию файлов.

Плюсы:

- единая схема данных;
- контроль обязательных полей;
- предсказуемая генерация маршрутов;
- возможность группировать переводы по общему slug.

Этот вариант стал основой итоговой модели.

---

# 5. Выбранный подход

![Гибридная модель коллекции и локалей](../content-scaling-architecture-solution.svg)

Была реализована гибридная схема:

1. Все статьи находятся в одной коллекции `post`.
2. Язык определяется директорией локали.
3. Один и тот же материал во всех локалях использует общий basename файла.
4. Связь переводов строится по общей части slug.
5. Навигация и списки статей собираются из коллекции автоматически.

Пример структуры:

```text
src/content/post/
  ru/astrowind_pf_blog_content_scaling_architecture.md
  en/astrowind_pf_blog_content_scaling_architecture.md
  uk/astrowind_pf_blog_content_scaling_architecture.md
  it/astrowind_pf_blog_content_scaling_architecture.md
```

В такой модели локали остаются изолированными на уровне контента, но маршрутизация, сортировка, теги и SEO контролируются централизованно.

---

# 6. Детали реализации

## 6.1. Схема коллекции

В `src/content/config.ts` коллекция `post` описывает обязательные поля и не даёт случайно увести статьи в произвольный формат frontmatter.

Это убирает часть ошибок ещё на этапе сборки.

## 6.2. Нормализация slug и тегов

При загрузке постов проект вычисляет slug, permalink, category и tags централизованно через `src/utils/blog.ts`.

Это даёт два эффекта:

- одинаковые правила для всех локалей;
- автоматическую нормализацию тегов для маршрутов.

## 6.3. Группировка переводов

Локализованные версии статьи объединяются по общей части slug:

```ts
const commonSlugs = [...new Set(posts.map((post) => post.slug.split('/')[1]))];
```

После этого можно строить списки и маршруты для каждой локали без отдельной ручной конфигурации.

## 6.4. Автоматическое попадание в блог

Список последних статей и основной блог используют данные коллекции напрямую.

Это означает, что новая статья:

- появляется в общем списке блога автоматически;
- участвует в блоке последних материалов по дате публикации;
- попадает на страницы тегов без ручного добавления в меню.

---

# 7. Компромиссы

У выбранной модели есть и цена:

1. Немного усложняется build-time логика.
2. Нужно соблюдать единое соглашение по именам файлов между локалями.
3. Появляется дисциплина в оформлении frontmatter.

Но это контролируемая сложность. Она значительно дешевле, чем постоянная ручная синхронизация контента и маршрутов.

---

# 8. Результат

После перехода на такую модель были получены практические преимущества:

1. Добавление нового языка не требует изменения архитектуры блога.
2. Структура URL остаётся предсказуемой.
3. Теги и категории собираются централизованно.
4. Навигация синхронизируется автоматически.
5. Количество точек отказа уменьшается.

В итоге масштабирование начинает зависеть в первую очередь от числа статей, а не от роста архитектурной сложности.

---

# 9. Выводы

Из этой задачи получились несколько устойчивых выводов:

1. Масштабирование контента связано со структурой данных, а не только с объёмом текста.
2. Дублирование разрушает поддерживаемость быстрее, чем кажется на старте проекта.
3. Типизированный frontmatter полезен даже для Markdown-проектов.
4. Группировка переводов по общему slug даёт устойчивую модель локализации.
5. Статический сайт может масштабироваться предсказуемо, если контентная архитектура выбрана правильно.

Именно поэтому архитектурные решения для блога лучше принимать исходя не из текущего количества статей, а из того, как проект будет расширяться дальше.
