# 1. Контекст

У проєктах на основі Markdown (документація, технічні блоги, портфоліо, внутрішні бази знань) навігація часто формується вручну. При масштабуванні структури — додаванні мов, розділів, вкладених категорій — підтримка меню стає окремим завданням.

При наявності кількох мов, повторно використовуваних компонентів та кількох типів сторінок (статті, кейси, документація) ручна синхронізація навігації перестає бути стійким рішенням.

Мета — сформувати архітектуру, при якій структура проєкту стає джерелом істини, а навігація та допоміжні дані генеруються автоматично.

---

# 2. Проблема

Основні симптоми:

- Дублювання структури в конфігураціях.
- Розбіжність між файловою системою та меню.
- Ручна синхронізація при додаванні мов.
- Зростання кількості помилок при рефакторингу.
- Втрата консистентності URL.

У багатомовних проєктах проблема посилюється: одна і та ж ієрархія має підтримуватися в декількох директоріях.

---

# 3. Обмеження

- Markdown залишається основним джерелом контенту.
- Не допускається ручне редагування навігації в декількох місцях.
- Структура має бути масштабованою.
- Підтримка SEO (зрозумілі URL, стабільність посилань).
- Мінімальна залежність від зовнішніх плагінів.
- Можливість інтеграції AI в процес генерації та перевірки.

---

# 4. Розглянуті варіанти

## Варіант 1: Повністю ручна конфігурація

Навігація описується в YAML/JSON конфігурації.

Плюси:
- Повний контроль порядку.

Мінуси:
- Дублювання.
- Висока ймовірність розсинхронізації.
- Низька масштабованість.

Рішення відхилено через високе операційне навантаження.

---

## Варіант 2: Автоматична генерація на основі файлової структури

Меню будується шляхом сканування директорій.

Плюси:
- Немає дублювання.
- Файлова система — джерело істини.

Мінуси:
- Втрата контролю над порядком.
- Обмежена гнучкість.

Рішення прийнято частково.

---

## Варіант 3: Метадані у Frontmatter + генератор навігації

Кожен Markdown‑файл містить структурні параметри:

- order
- section
- lang
- slug

Генератор формує дерево на основі метаданих.

Плюси:
- Контроль порядку.
- Підтримка багатомовності.
- Розширюваність.

Мінуси:
- Потрібен додатковий шар обробки.

Обрано як основний підхід.

---

# 5. Обраний підхід

Архітектура складається з трьох рівнів:

1. Файлова структура — базова ієрархія.
2. Frontmatter — декларативні метадані.
3. Генератор навігації — скрипт, що формує структуру.

Файлова система визначає розділи, а метадані — порядок і логічне групування.

Структура проєкту:

```
content/
  en/
    blog/
    docs/
  ru/
    blog/
    docs/
scripts/
  generate-navigation.ts
```

Генератор:
- Сканує директорії.
- Читає frontmatter.
- Будує дерево.
- Експортує JSON.

---

# 6. Деталі реалізації

## 6.1. Формат Frontmatter

```
---
title: "Стаття"
order: 2
section: "архітектура"
lang: "uk"
slug: "авто-навігація"
---
```

## 6.2. Алгоритм генерації

1. Рекурсивне сканування каталогу.
2. Групування за мовами.
3. Групування за section.
4. Сортування за order.
5. Генерація структури:

```
{
  "uk": {
    "архітектура": [ ... ]
  }
}
```

## 6.3. Інтеграція в збірку

- Скрипт запускається перед збіркою.
- JSON імпортується в layout.
- Меню формується динамічно.

## 6.4. AI у workflow

- Перевірка консистентності метаданих.
- Пошук відсутніх slug.
- Аналіз конфліктів order.

AI використовується як інструмент валідації, а не джерело структури.

---

# 7. Компроміси (Trade-offs)

| Рішення | Перевага | Обмеження |
|----------|--------------|-------------|
| Автогенерація | Немає дублювання | Додатковий шар обробки |
| Frontmatter | Контроль порядку | Вимагає дисципліни |
| JSON-експорт | Чиста архітектура | Необхідна синхронізація при збірці |

Головний компроміс — збільшення складності інфраструктури заради стійкості та масштабованості.

---

# 8. Результат

- Структура стала джерелом істини.
- Додавання нової мови не вимагає зміни конфігурації.
- Виключено ручне оновлення меню.
- Знижено кількість помилок при рефакторингу.
- Підвищена передбачуваність URL.

Навігація масштабується лінійно з ростом проєкту.

---

# 9. Висновки та уроки

1. Файлова структура має бути декларативною.
2. Метадані мають контролювати логіку, а не конфігурації.
3. Автоматизація виправдана при зростанні проєкту.
4. AI корисний як інструмент перевірки, але не як джерело архітектури.
5. Стійкість важливіша за локальну зручність.

Автоматизація навігації — не оптимізація, а перехід до керованої архітектури Markdown‑проєктів.
