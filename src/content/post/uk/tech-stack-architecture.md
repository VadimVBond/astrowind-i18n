# 1. Контекст

![Обкладинка технологічного стеку](../tech-stack-architecture-cover.svg)

У роботі використовується стек, орієнтований на створення статичних і гібридних сайтів, документаційних систем і backend-рішень на Python. Основні напрями включають Astro для frontend-оркестрації, MkDocs для документаційних проєктів, Python-фреймворки для API та AI-інструменти як допоміжний шар для аналізу й прискорення інженерного процесу.

Мета цього стеку полягає не в кількості технологій самій по собі. Головне завдання — архітектурна керованість: передбачуване масштабування, повторне використання компонентів і структура, що залишається цілісною зі зростанням складності проєкту.

---

# 2. Проблема

Більшість малих і середніх проєктів стикаються з однаковими труднощами:

- фрагментована структура коду;
- слабка стратегія повторного використання;
- відсутність системної багатомовності;
- змішування контенту, логіки та представлення;
- втрата керованості зі зростанням проєкту.

Окремо постає завдання інтегрувати AI-інструменти без втрати архітектурної дисципліни. Стек має залишатися модульним, підтримувати різні масштаби проєктів і не змушувати змінювати платформу щоразу, коли змінюються вимоги.

---

# 3. Обмеження

Вибір технологій відбувався з урахуванням таких обмежень:

1. Низька вартість інфраструктури.
2. Можливість деплою на edge-платформи та статичний хостинг.
3. Мінімальна залежність від важких CMS.
4. Чітке розділення frontend і backend.
5. Можливість використовувати AI без передачі чутливих даних.
6. Ручний контроль структури проєкту замість прихованих абстракцій.

Стек має бути керованим на рівні архітектури, а не лише окремих фреймворків.

---

# 4. Розглянуті варіанти

## 4.1 Повноцінний SPA-підхід

Розглядалися Next.js та подібні рішення.

Переваги:

- висока гнучкість;
- зріла екосистема.

Причини відмови:

- надмірна складність для більшості проєктів;
- вищі інфраструктурні витрати;
- SEO-ризики при некоректній конфігурації.

## 4.2 CMS-підхід

Розглядалися WordPress та подібні системи.

Переваги:

- швидкий старт;
- велика кількість готових плагінів.

Причини відмови:

- слабкий архітектурний контроль;
- обмежена масштабованість структури;
- вищий ризик технічного боргу.

## 4.3 Чистий backend плюс шаблони

Переваги:

- повний контроль над серверною логікою.

Причини відмови:

- дублювання логіки;
- слабке повторне використання UI;
- складності з масштабуванням frontend-частини.

---

# 5. Обраний підхід

Було обрано гібридний архітектурний стек:

- Astro як основний frontend-оркестратор;
- MkDocs для документаційних і структурованих контентних проєктів;
- Python на Django Ninja, FastAPI та Flask для API і серверної логіки;
- AI-інструменти як допоміжний інженерний шар.

Ключова ідея полягає в розділенні відповідальності.

| Рівень | Інструмент | Роль |
| --- | --- | --- |
| UI та контент | Astro | Статична генерація, SEO, композиція сторінок |
| Документація | MkDocs | Структурований контент і навігація |
| API | Django Ninja / FastAPI | Легкий типізований backend |
| Легкі сервіси | Flask | Ізольовані мікросервіси та прототипи |
| AI | Gemini / Copilot / ChatGPT | Підтримка аналізу, чернеток і перевірки |

# 6. Деталі реалізації

## 6.1 Frontend на Astro

Базова структура виглядає так:

```text
src/
  components/
  layouts/
  pages/
  libs/
  i18n/
public/
```

Ключові принципи:

- компоненти не залежать від конкретних сторінок;
- багатомовність будується через структуру папок і словники;
- повторно використовувані блоки виносяться в `libs/`.

## 6.2 Стратегія багатомовності

Використовуються:

- структурне розділення мов (`/en/`, `/ru/`, `/uk/`, `/it/`);
- централізовані словники;
- явна маршрутизація;
- SEO-орієнтована структура URL.

Мовний шар розглядається як частина архітектури, а не як надбудова після реалізації.

## 6.3 Backend на Python

### Django Ninja / FastAPI

Використовуються для:

- JSON API;
- адміністративних сервісів;
- інтеграції з базами даних.

Вибір обумовлений:

- продуктивністю;
- типізацією;
- передбачуваною структурою проєкту.

### Flask

Використовується для:

- легких сервісів;
- прототипування;
- ізольованих мікросервісів.

## 6.4 AI у workflow

AI розглядається як підсилювач якості, а не як джерело архітектури.

Його ролі включають:

- генерацію чернеток і рутинних заготовок;
- посилення архітектурного аналізу;
- уточнення технічних текстів і формулювань.

AI не визначає структуру проєкту. Він пришвидшує роботу й додає додатковий рівень перевірки.

## 6.5 Повторне використання компонентів

Компоненти проєктуються як:

- незалежні;
- не прив'язані до одного конкретного проєкту;
- ізольовані за стилями;
- придатні до винесення в окремі бібліотеки.

Мета полягає у формуванні внутрішньої бази повторно використовуваних компонентів.

# 7. Компроміси

## 7.1 Astro проти SPA

Плюси:

- висока продуктивність;
- прямий контроль над HTML;
- SEO-дружній результат.

Мінуси:

- менше інтерактивності за замовчуванням.

Рішення: використовувати islands architecture лише там, де інтерактивність справді потрібна.

## 7.2 Багатомовність через структуру папок

Плюси:

- явність;
- прозорість для SEO.

Мінуси:

- більша кількість файлів;
- вищий обсяг підтримки.

## 7.3 Розділення backend і frontend

Плюси:

- незалежний розвиток;
- краща масштабованість.

Мінуси:

- додаткова координація API.

## 7.4 AI-інтеграція

Плюси:

- швидша розробка;
- краща якість чернеток і формулювань.

Мінуси:

- ризик поверхневих рішень;
- постійна потреба в інженерній перевірці.

---

# 8. Результат

У результаті формується стек, який:

- дозволяє швидко запускати проєкти;
- залишається передбачуваним у супроводі;
- масштабується від лендингів до складних структур;
- лишається керованим на архітектурному рівні;
- підтримує повторне використання компонентів;
- добре поєднується з сучасним AI-assisted workflow.

Проєкти при цьому зберігають високу продуктивність, чисту структуру, SEO-орієнтовану організацію та готовність до подальшого розширення.

---

# 9. Висновки

1. Технологічний стек слід обирати за архітектурною сумісністю, а не за трендовістю.
2. Простота супроводу важливіша за ширину екосистеми сама по собі.
3. Багатомовність, SEO та повторне використання потрібно закладати в структуру з самого початку.
4. AI покращує процес лише тоді, коли вже існує архітектурна дисципліна.
5. Сильний стек зменшує вартість майбутніх змін і дозволяє проєкту зростати без хаотичного ускладнення.

Саме тому хороший стек визначається не кількістю інструментів, а узгодженістю їхніх ролей усередині системи.
