# 1. Контекст

![Статичний сайт](https://images.unsplash.com/photo-1554130916-c2aebf8c8770)

Мета — створення портфоліо-проєкту та інженерного блогу без використання CMS.

Вимоги:

- Повна статична генерація
- Підтримка кількох мов
- Контент на Markdown
- Контроль над структурою URL
- Мінімізація runtime-логіки
- Передбачувана SEO-архітектура

Стек:

- MkDocs для документаційних розділів
- Astro для портфоліо та блогової частини
- Tailwind / Bootstrap залежно від завдання

Проєкт проєктувався як розширюваний та CDN-сумісний.

---

# 2. Проблема

![CMS vs Static](https://images.unsplash.com/photo-1559526324-593bc073d938)

Більшість багатомовних сайтів будуються на CMS:

- WordPress + плагіни
- Headless CMS
- Strapi / Contentful

Проблема такого підходу:

- Надмірність для портфоліо
- Ускладнення інфраструктури
- API-залежність
- Підвищена точка відмови
- Runtime-залежність

Завдання: реалізувати багатомовність без бази даних та без адміністративної панелі.

---

# 3. Обмеження

![Обмеження](https://images.unsplash.com/photo-1581091870629-33b3c9f2c55e)

- Повна статична генерація (SSG)
- Відсутність сервера додатків
- SEO-сумісність
- Можливість хостингу на CDN
- Контроль canonical та hreflang
- Мінімізація дублювання структури

# 4. Розглянуті варіанти

## Варіант A — Папкова сегрегація

```
/en/
/ru/
/uk/
```

Плюси:
- Проста логіка
- Явне розділення

Мінуси:
- Дублювання структури
- Підвищений ризик розсинхронізації
- Зростання складності підтримки

Відмова: висока вартість супроводу при масштабуванні.

## Варіант B — Headless CMS

Плюси:
- Управління через UI
- Централізація даних

Мінуси:
- Ускладнення архітектури
- API-залежність
- Втрата статичної простоти
- Надмірність для портфоліо

Відмова: невиправдана складність.

## Варіант C — Content collections з lang-полем (обраний)

```
src/content/blog/article.ru.md
src/content/blog/article.en.md
```

Плюси:
- Єдиний layout
- Спільний slug
- Централізований SEO-контроль
- Просте масштабування

Мінуси:
- Потрібна дисципліна в naming-конвенціях
- Немає автоматичної синхронізації перекладів

![Контент колекції](https://images.unsplash.com/photo-1590608897129-79cfef5b9f3f)

# 5. Обраний підхід

Ключова ідея — мова як атрибут контенту, а не як структурна папка.

Кожна стаття:
- має єдиний slug
- відрізняється тільки `lang`
- генерується в просторі `/ru/slug` (або `/uk/slug` тощо)

Це дозволяє:
- централізувати SEO
- управляти hreflang
- не дублювати layout
- зберегти статичну простоту

![SEO та hreflang](https://images.unsplash.com/photo-1537432376769-00a4c61f2010)

# 6. Деталі реалізації

## Структура

```bash
src/
  content/
    blog/
      multilingual-architecture.ru.md
      multilingual-architecture.en.md
```

## Frontmatter

```yaml
lang: ru # або uk тощо
slug: multilingual-architecture
```

## Генерація маршрутів
- Отримання всіх статей
- Групування за slug
- Генерація alternate-посилань

## SEO
- Canonical для кожної мовної версії
- Автоматична генерація hreflang
- Виключення draft з індексації

## Навігація
- Автоматична генерація на основі content collections
- Без ручного дублювання структури

![Маршрути та навігація](https://images.unsplash.com/photo-1557804506-669a67965ba0)

# 7. Компроміси (Trade-offs)

1. Немає адміністративного інтерфейсу
2. Немає live-перекладів
3. Немає runtime-перемикання мови
4. Потрібна дисципліна структури

Архітектура спрощує інфраструктуру, але вимагає інженерної суворості.

# 8. Результат

- Повністю статичний сайт
- SEO-коректна багатомовність
- Мінімальна точка відмови
- Прогнозована масштабованість

Підтримує:
- 50+ статей
- 3+ мови
- розширення до документаційного розділу

# 9. Висновки та уроки

Багатомовність не потребує CMS.

Якщо:
- обсяг контенту обмежений
- немає вимог до runtime-редагування
- важлива передбачуваність архітектури

Статичний підхід виявляється стійкішим.

Ключовий фактор — дисципліна в організації контенту та slug-конвенціях.
