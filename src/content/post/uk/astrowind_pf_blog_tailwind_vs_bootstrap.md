# Коли Tailwind краще за Bootstrap (і навпаки)

## 1. Контекст

У межах розробки проєктів на Astro з багатомовною архітектурою, компонентами для повторного використання та суворими вимогами до продуктивності виникає необхідність системного вибору CSS-підходу.

Варіанти — Tailwind (utility-first) та Bootstrap (component-based framework). Обидва інструменти є зрілими, але архітектурно орієнтовані на різні завдання.

Вибір впливає на:

- структуру проєкту
- повторне використання компонентів
- об'єм CSS
- швидкість збірки
- масштабованість дизайн-системи
- довгострокову підтримку

Завдання — визначити, за яких умов кожен інструмент є раціональним архітектурним рішенням.

---

## 2. Проблема

Типова помилка — вибирати інструмент за особистими вподобаннями.

У реальних проєктах критерії інші:

- чи потрібна сувора дизайн-система
- чи буде проєкт рости
- чи передбачається кастомний UI
- чи важлива швидкість прототипування
- чи є обмеження щодо SEO та performance
- скільки розробників працюватиме над кодовою базою

Bootstrap та Tailwind по-різному вирішують ці завдання.

---

## 3. Обмеження

У межах Astro-проєкту враховувалися такі обмеження:

- Статична генерація (SSG)
- Багатомовність
- Поділ layout / components / blocks
- Можливість часткового повторного використання UI між проєктами
- Мінімізація CSS-бандла
- Відсутність надлишкового JavaScript

Додатково:

- Компоненти мають бути ізольовані
- Має підтримуватися темізація (dark/light)
- Потрібна гнучкість для кастомних лендінгів

---

## 4. Розглянуті варіанти

### Варіант 1 — Bootstrap як основний UI-фреймворк

Плюси:

- Швидкий старт
- Готові компоненти
- Консистентна сітка
- Передбачувана структура

Мінуси:

- Надлишковий CSS
- Обмежена гнучкість кастомізації
- Стилі за замовчуванням вимагають перевизначення
- Компонентна логіка може конфліктувати з Astro-архітектурою

---

### Варіант 2 — Tailwind як utility-first система

Плюси:

- Мінімальний підсумковий CSS (через purge)
- Висока гнучкість
- Хороша інтеграція з компонентною архітектурою
- Зручність темізації

Мінуси:

- Підвищена щільність класів у розмітці
- Вимагає дисципліни
- Потрібна власна дизайн-система

---

### Варіант 3 — Гібридна модель

Bootstrap для сітки та базових компонентів.
Tailwind для кастомної стилізації.

Результат — дублювання відповідальності та зростання складності.

Відмова від варіанта через архітектурну неоднозначність.

---

## 5. Обраний підхід

Підхід визначався типом проєкту.

### Tailwind використовується коли:

- Проєкт вимагає кастомного дизайну
- Є власна дизайн-система
- Потрібна висока гнучкість
- Потрібен мінімальний CSS-бандл
- Компоненти мають бути максимально придатними для повторного використання

### Bootstrap використовується коли:

- Потрібен швидкий MVP
- Дизайн стандартний
- Проєкт невеликий
- Пріоритет — швидкість розробки
- Немає ресурсів на побудову власної системи

Таким чином, вибір став контекстно-залежним.

---

## 6. Деталі реалізації

### Архітектура з Tailwind

```
src/
 ├── layouts/
 ├── components/
 ├── blocks/
 ├── styles/
 │    └── global.css
 └── config/
      └── tailwind.config.js
```

Ключові принципи:

- Конфігурація design tokens у tailwind.config.js
- Використання @apply тільки в шарі abstraction
- Темізація через class="dark"
- Поділ layout та UI-компонентів

Приклад design token:

```js
// tailwind.config.js
export default {
  theme: {
    extend: {
      colors: {
        primary: '#1e293b'
      }
    }
  }
}
```

Результат — керована дизайн-система.

---

### Архітектура з Bootstrap

```
src/
 ├── layouts/
 ├── components/
 ├── assets/
 │    └── bootstrap.scss
```

Ключові кроки:

- Використання SCSS-версії
- Відключення непотрібних модулів
- Кастомізація змінних

Приклад:

```scss
$primary: #1e293b;
@import "bootstrap/scss/bootstrap";
```

Проте підсумковий CSS залишається істотно більшим.

---

## 7. Компроміси (Trade-offs)

| Критерій | Tailwind | Bootstrap |
|----------|-----------|------------|
| Розмір CSS | Мінімальний | Значний |
| Швидкість старту | Середня | Висока |
| Кастомізація | Висока | Обмежена |
| Підтримка дизайн-системи | Гнучка | Жорстка |
| Підтримка коду | Вимагає дисципліни | Передбачувана |
| SEO (опосередковано через performance) | Краще | Гірше |

Ключовий компроміс:

Tailwind вимагає архітектурного мислення.
Bootstrap знижує поріг входу, але обмежує масштабованість.

---

## 8. Результат

У межах Astro-проєктів з довгостроковою підтримкою та багатомовною структурою перевагу віддано Tailwind.

Причини:

- Контроль над CSS
- Можливість побудови власної дизайн-системи
- Краща інтеграція з компонентною архітектурою
- Мінімізація технічного боргу

Bootstrap залишається інструментом для:

- Прототипів
- Адмін-панелей
- Внутрішніх інструментів

---

## 9. Висновки та уроки

1. Немає універсального рішення.
2. Вибір має залежати від масштабу та горизонту проєкту.
3. Для довгострокової архітектури utility-first модель стійкіша.
4. Для швидких рішень компонентний фреймворк раціональний.
5. Гібридний підхід збільшує складність.

Фреймворк стилів — це не візуальний вибір. Це архітектурне рішення, яке впливає на підтримку, масштабованість та структуру проєкту.
