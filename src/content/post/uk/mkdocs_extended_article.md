## 1. Контекст
![MkDocs структура проєкту](../../../assets/images/posts/mkdocs/mkdocs-folder-structure.png)
MkDocs традиційно використовується для документації проєктів. Його структура (`docs/`, `mkdocs.yml`) орієнтована на лінійний контент, що обмежує кастомізацію фронтенду та мультимовного контенту. З ростом вимог до інтерактивних сайтів з'явилася необхідність розглянути MkDocs для більш широких завдань.

## 2. Проблема
![Проблема інтеграції](../../../assets/images/posts/mkdocs/mkdocs-problem.png)
Необхідно використовувати MkDocs для проєктів, що виходять за межі документації, зберігаючи:
- Статичність та простоту деплою
- Мультимовність
- Можливість динамічних компонентів
- Розподіл глобальних та локальних стилів

Обмеження: MkDocs не надає повноцінної компонентної системи.

## 3. Обмеження
![Обмеження MkDocs](../../../assets/images/posts/mkdocs/mkdocs-limitations.png)
- Немає нативної динаміки
- Обмежені можливості шаблонів (Jinja2 + Markdown)
- SEO та URL залежать від генератора
- Складність навігації у великих проєктах

## 4. Розглянуті варіанти
![Альтернативні рішення](../../../assets/images/posts/mkdocs/mkdocs-alternatives.png)
1. React/Next.js — SSR та компоненти, відмова через надмірну складність.
2. Astro + Markdown — гнучка генерація, відмова через складність інтеграції.
3. MkDocs з кастомними include та блоками — обраний підхід: зберігає Markdown, дозволяє інтерактивні елементи та мультимовність.

## 5. Обраний підхід
![Архітектура MkDocs](../../../assets/images/posts/mkdocs/mkdocs-architecture-diagram.png)
- `overrides/base.html` — структура сторінки
- `overrides/includes/blocks/` — UI-блоки для повторного використання
- `overrides/includes/headers/` та `footers/` — перемикачі мов та тем
- `docs/` — контент за мовами (`en/`, `ru/`, `uk/`)

Контент відокремлений від представлення, блоки можна підключати без модифікації ядра.

## 6. Деталі реалізації
1. Мультимовність: nav формується скриптом, include-блоки з прапорцями.
2. Компоненти: окремі HTML-файли з локальними стилями та JS, глобальні стилі в assets/css.
3. Інтерактивні елементи: слайдери Swiper, фільтри з JSON.
4. SEO: окремі URL для кожної мови, метатеги через base.html.

## 7. Компроміси (Trade-offs)
![Компроміси](../../../assets/images/posts/mkdocs/mkdocs-tradeoffs.png)
| Фактор | Плюси | Мінуси |
|--------|-------|--------|
| Деплой | Статичний сайт | Обмежена динаміка |
| Мультимовність | Чітка структура | Скрипт генерації nav потрібен |
| Компоненти | Повторне використання | Немає повноцінної компонентної системи |
| SEO | Ручне налаштування метатегів | Немає SSR |
| Продуктивність | Легкі сторінки | JS для інтерактивності збільшує розмір |

## 8. Результат
![Результат реалізації](../../../assets/images/posts/mkdocs/mkdocs-result.png)
- Підтримка мультимовності та SEO
- Компоненти для повторного використання
- Інтерактивні елементи без складного білду
- Масштабованість через `overrides/includes`

## 9. Висновки та уроки
![Висновки](../../../assets/images/posts/mkdocs/mkdocs-lessons.png)
- MkDocs підходить для контентних сайтів з інтерактивністю
- Відокремлення контенту, шаблонів та блоків є критичним для розширюваності
- Обмеження динаміки компенсуються мінімальним JS та JSON
- Підтримуваність залежить от дисципліни в організації include-блоків
